Admin Guide

The admin panel lets you manage Nostr identities, users, sessions, and access assignments. It is protected by NIP-98 authentication — you need a NIP-07 browser extension and your pubkey must be in the ADMIN_PUBKEYS allowlist.

Authentication

Admin access uses NIP-98 HTTP Auth, a Nostr-native authentication mechanism. Each API request is signed with your Nostr key via a NIP-07 browser extension.

Prerequisites

  • A NIP-07 browser extension installed (e.g., nos2x, Alby)
  • Your hex pubkey added to the ADMIN_PUBKEYS environment variable

Setting up admin access

  1. Get your hex pubkey from your NIP-07 extension
  2. Add it to your .env file: 
    ADMIN_PUBKEYS=your_hex_pubkey_here
    Multiple admins can be comma-separated: 
    ADMIN_PUBKEYS=pubkey1,pubkey2,pubkey3
  3. Restart the server
  4. Navigate to /admin and click "Connect with Nostr"
  5. Your extension will prompt you to share your public key and sign auth events

How NIP-98 auth works

  • Each admin API request includes an Authorization: Nostr <base64> header
  • The header contains a signed kind 27235 event with the request URL and HTTP method
  • The server verifies the signature, timestamp (within 60 seconds), URL, method, and pubkey
  • No sessions or tokens — every request is independently verified

Admin Panel Layout

After authenticating, the admin panel shows a sidebar with four sections:

  • Sessions — All active NIP-46 connections across all users
  • Users — Registered OAuth users
  • Secret Keys — Nostr identities in the pool
  • Assignments — User-to-identity access grants

Sessions

The Sessions table shows all active NIP-46 connections across all users. Each row displays the user email, assigned identity, client public key, relay URL, and last activity timestamp.

  • Use the Revoke button to immediately disconnect a client session
  • Revoked clients will need to re-authenticate to reconnect
  • Sessions represent active Nostr client connections (e.g., from Habla, Amethyst, or other NIP-46 clients)

Managing Identities (Secret Keys)

Adding an identity

  • In the Secret Keys section, use the inline form at the top
  • Paste a Nostr nsec (bech32 format, starts with nsec1...)
  • Optionally add a label (e.g., "Blog", "Personal", "Work")
  • The nsec is immediately encrypted with AES-256-GCM
  • Encryption uses a key derived from MASTER_KEY via HKDF-SHA256
  • The plaintext nsec is never stored — only the encrypted blob and nonce

Viewing identities

  • The table shows: public key (hex, truncated), label, and active connection count
  • The pubkey can be shared — it's the public identifier for the Nostr identity

Deleting an identity

  • Removes the encrypted nsec from the database
  • Disconnects all active NIP-46 connections using that identity
  • Removes all user assignments for that identity
  • This action is irreversible — make sure you have the nsec backed up elsewhere

Managing Users

  • Users are created automatically when they first sign in via OAuth
  • The table shows: avatar, email, OAuth provider, and join date
  • Users cannot be created manually — they must authenticate via OAuth

Identity Assignments

Creating an assignment

  • In the Assignments section, use the inline form to select a user, identity, and duration
  • Available durations: 1 day, 1 week, 1 month, 6 months, or 1 year
  • The user can now select this identity when connecting via NIP-46

How assignments work

  • Users can only see and use identities they're assigned to
  • When a user authenticates via OAuth and reaches the identity picker, only their assigned (non-expired) identities appear
  • Assignment expiration is enforced at signing time — the bunker checks before signing each event
  • A background task runs every 5 minutes to clean up expired assignments
  • The Status column shows Active or Expired badges

Revoking access

  • Delete the assignment to immediately revoke access
  • The user will no longer see the identity in their picker
  • Existing NIP-46 connections using that identity will fail at next signing attempt

API Endpoints

All admin endpoints require NIP-98 authentication.

Method Endpoint Description
GET/api/admin/connectionsList all sessions
DELETE/api/admin/connections/{id}Revoke a session
GET/api/admin/identitiesList all identities
POST/api/admin/identitiesAdd an identity
DELETE/api/admin/identities/{id}Delete an identity
GET/api/admin/usersList all users
GET/api/admin/assignmentsList all assignments
POST/api/admin/assignmentsCreate an assignment
DELETE/api/admin/assignments/{id}Delete an assignment

Common Admin Tasks

Onboarding a new user

  1. Have the user sign in via any OAuth provider
  2. Find them in the Users section
  3. Switch to Assignments and create access grants for the identities they should use
  4. Share the bunker connection string (domain or bunker URI) with them

Rotating an identity

  1. Generate a new nsec externally
  2. Add the new identity in Secret Keys
  3. Reassign affected users to the new identity
  4. Delete the old identity (disconnects all sessions)

Emergency revocation

  1. To block a specific user: delete their assignments
  2. To disconnect a specific client: revoke it in Sessions
  3. To kill all access to an identity: delete the identity entirely
