Architecture

The Nostr OAuth Signer bridges traditional OAuth authentication with Nostr's cryptographic identity model. It combines an OAuth gateway, an encrypted key vault, and a NIP-46 bunker.

System Overview

Three core components:

  1. OAuth Gateway — Handles authentication with Google, GitHub, Microsoft, Apple. Creates user sessions.
  2. Encrypted Key Vault — Stores Nostr private keys (nsecs) encrypted with AES-256-GCM. Keys derived via HKDF from a master key.
  3. NIP-46 Bunker — Implements the Nostr Connect protocol for remote signing. Clients connect via relays.

NIP-46 Bunker Protocol

NIP-46 (Nostr Connect) allows Nostr clients to request signing operations from a remote signer (bunker) without the client ever seeing the private key.

Supported methods:

  • connect — Establish a new connection
  • sign_event — Sign a Nostr event
  • get_public_key — Get the identity's public key
  • get_relays — Get relay list
  • nip44_encrypt — NIP-44 encryption
  • nip04_encrypt — NIP-04 encryption (legacy)

How clients connect: Clients use a bunker URI containing the bunker's public key and relay URLs. When connecting, the bunker responds with an auth URL that redirects the user through OAuth.

Connection Flow

Step-by-step flow when a Nostr client (e.g., Habla) connects:

  1. Client requests connection — Sends NIP-46 connect request to bunker pubkey via relay
  2. Bunker responds with auth URL — Returns URL pointing to web service (e.g., https://web.example.com/auth/abc123)
  3. User authenticates — Browser opens auth URL, redirects to OAuth provider, user logs in
  4. Session created — OAuth callback creates/finds user, sets session cookie
  5. Identity selection — Web service shows identity picker with user's assigned identities
  6. Approval sent — Web service sends encrypted approval event to bunker via relay
  7. Connection established — Bunker stores connection, replies with ack to client
  8. Signing (steady state) — Client sends sign_event, bunker decrypts nsec, signs, zeroizes

Key Encryption Model

Master key: A 32-byte random key stored as an environment variable (MASTER_KEY). Never stored in the database.

Key derivation: Each identity's encryption key is derived from the master key using HKDF-SHA256:

identity_key = HKDF-SHA256(salt=identity_id, key=MASTER_KEY)

Encryption: Each nsec is encrypted with AES-256-GCM using a random 12-byte nonce. The database stores: encrypted blob + nonce + public key. The plaintext nsec is never persisted.

Signing flow:

  1. Client sends sign_event request
  2. Bunker retrieves encrypted nsec from database
  3. Derives decryption key: HKDF(MASTER_KEY, identity_id)
  4. Decrypts nsec in memory
  5. Signs the event
  6. Zeroizes (securely erases) the plaintext nsec from memory
  7. Returns signed event

Database Schema

Overview (not full DDL):

users — OAuth user accounts

  • id (UUID), oauth_provider, oauth_sub, email, avatar_url, created_at
  • Unique on (provider, sub)

identities — Nostr keypairs (encrypted)

  • id (UUID), encrypted_nsec (blob), nonce (blob), pubkey (unique), label, created_at

user_identity_assignments — Maps users to identities with expiration

  • id (UUID), user_id → users, identity_id → identities, expires_at, created_at

connections — Active NIP-46 sessions

  • id, user_id, client_pubkey, relay_url, created_at, last_used_at

sessions — OAuth web sessions

  • token (primary key), user_id, expires_at

pending_auth — Transient connection requests

  • request_id, client_pubkey, relay_url, nip46_id, created_at, expires_at

Security Properties

Key security guarantees:

  • NSECs are encrypted at rest, decrypted only in bunker memory during signing, then zeroized
  • In split architecture, NSECs never leave the LAN
  • The web service never sees or handles signing keys
  • Assignment expiration is enforced at signing time (not just at connection time)
  • OAuth sessions are separate from NIP-46 connections — compromising one doesn't compromise the other
  • Admin endpoints only exist on the bunker, which is LAN-only
  • The only message between web service and bunker is a single encrypted approval event (NIP-44)
