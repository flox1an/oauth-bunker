Production Deployment

There are two deployment models: monolithic (single server) and split (bunker + web service). Both can use Cloudflare Tunnel to securely expose your service to the internet without opening ports.

Deployment Models

Monolithic runs everything on a single server — the bunker, web UI, OAuth flows, and admin API all in one process. This is simpler to set up and suitable for personal use or small teams.

Split separates the bunker (which holds signing keys) from the internet-facing web service. This is more secure for production because a compromised web server never has access to NSECs. See Split Architecture below for details.

Monolithic Deployment

  • A single server runs the bunker, web UI, OAuth, and admin API in one process.
  • Suitable for personal use or small teams.
  • Use a reverse proxy (such as Cloudflare Tunnel) to expose only the routes that need to be public.
  • Admin routes (/api/admin/*) should stay LAN-only — never expose them to the internet.

Cloudflare Tunnel Setup

Cloudflare Tunnel lets you expose your local server to the internet without opening firewall ports. It creates an outbound-only connection from your server to Cloudflare's edge.

Install cloudflared

brew install cloudflared

Create the tunnel

cloudflared tunnel login
cloudflared tunnel create bunker-demo
cloudflared tunnel route dns bunker-demo bunker.yourdomain.com

Configure selective routing

Create ~/.cloudflared/config.yml to control which routes are exposed:

tunnel: <YOUR_TUNNEL_ID>
credentials-file: /home/user/.cloudflared/<YOUR_TUNNEL_ID>.json

ingress:
  # Block admin routes — keep these LAN-only
  - hostname: bunker.yourdomain.com
    path: /api/admin.*
    service: http_status:404

  # OAuth flow
  - hostname: bunker.yourdomain.com
    path: /auth.*
    service: http://localhost:3000

  # Auth popup (user picks identity after OAuth)
  - hostname: bunker.yourdomain.com
    path: /auth-popup.*
    service: http://localhost:3000

  # Web UI assets
  - hostname: bunker.yourdomain.com
    path: /assets.*
    service: http://localhost:3000

  # API endpoints needed by the web UI
  - hostname: bunker.yourdomain.com
    path: /api/me
    service: http://localhost:3000
  - hostname: bunker.yourdomain.com
    path: /api/select-identity
    service: http://localhost:3000
  - hostname: bunker.yourdomain.com
    path: /api/identities
    service: http://localhost:3000

  # NIP-89 discovery
  - hostname: bunker.yourdomain.com
    path: /.well-known/nostr.json
    service: http://localhost:3000

  # SPA index
  - hostname: bunker.yourdomain.com
    path: /
    service: http://localhost:3000

  # Block everything else
  - service: http_status:404

Update environment

Set PUBLIC_URL=https://bunker.yourdomain.com in your .env file. Then update the OAuth callback URLs in each provider's dashboard to match (e.g., https://bunker.yourdomain.com/auth/google/callback).

Run

cargo run
# In another terminal:
cloudflared tunnel run bunker-demo

Routes: Exposed vs Hidden

Route Exposed Purpose
/auth/* Yes OAuth initiation and callbacks
/auth-popup/* Yes Identity selection after OAuth
/api/me Yes Current user info for web UI
/api/select-identity Yes Identity picker for auth flow
/api/identities Yes List available identities
/.well-known/nostr.json Yes NIP-89 bunker discovery
/assets/* Yes Static frontend assets
/api/admin/* No Identity & user management
/api/connections No Connection management
NIP-46 bunker No Outbound relay connections only

Split Architecture

For production deployments, separate the bunker from the web service. This ensures that signing keys are never accessible from the internet-facing server.

  • Bunker (LAN-only): Holds NSECs and MASTER_KEY, handles NIP-46 signing, serves the admin API.
  • Web Service (internet-facing): Handles OAuth flows, session management, and the identity picker UI.
  • Shared SQLite DB (LAN): A single database on the LAN acts as the source of truth for both services.

Data Ownership

Data Bunker Web Service
NSECs / MASTER_KEY Yes No
Identity pubkeys + labels Yes (owns) Yes (read-only)
User-identity assignments Yes (owns) Yes (read-only)
OAuth credentials No Yes
Users / sessions No Yes (owns)
Pending auth requests Yes (owns) No
Connections Yes (owns) No

Security Properties

  • NSECs exist only in the bunker process on the LAN.
  • The bunker has zero inbound network connections — it connects outbound to relays only.
  • The web service never sees signing keys.
  • Only one relay message passes between services: an encrypted approval event (NIP-44).
  • Compromise of the web service leaks OAuth sessions but not NSECs.

Security Checklist

  • MASTER_KEY backed up securely (not in version control)
  • Admin routes (/api/admin/*) not exposed to the internet
  • HTTPS on the public-facing service
  • OAuth callback URLs match PUBLIC_URL exactly
  • Firewall rules prevent direct access to the bunker from the internet
  • Database file permissions restrict access
  • Regular backups of database and MASTER_KEY
